\section{Software, Hardware, and Networking}


\subsection{Software: Don't Self-Host}

\lesson{Don't self-host your exam software; the expected cost
  savings evaporates with other overheads.}

Because we had only one course actually trying to run CBTF-like exams
and a few others using PrairieLearn for formative assessments only,
we initially hosted PrairieLearn ourselves, since it was open source
and we didn't know if we'd need to make changes to integrate with our
campus systems.  In retrospect, this was a mistake.

Our IT staff is highly competent and were very cooperative.  However,
this represented a brand-new set of tools and
workflows for them, and their time and budget are tight.
And even with light usage, smooth operation during exams is mission-critical.
As soon as more courses began doing CBTF quizzes,
the manual scaling of Amazon EC2 (Elastic Compute Cloud) resources to
ensure sufficient capacity quickly became a nuisance, with IT staff
needing a 24-hour heads up before any scale-up period.  PrairieLearn
hosting would have made this transparent.

In addition, the open-source software does not include PrairieTest for
scheduling the CBTF.  Even with only one course to schedule, we
underestimated how useful PrairieTest would be, as we would discover
when we tried to do reservations ourselves.


\subsection{Hardware: Ancient Legacy Desktops}

\lesson{15-year-old desktop computers sitting in
  underutilized computer labs have a new lease on life.}

PrairieLearn exams require only a browser; any heavy lifting such as
code autograding is done on the server side by managing Docker
containers (another reason to have PrairieLearn host for you, although
as a caveat, a course requiring custom external grading containers costs more
than 2x per seat as one that doesn't).
Some of the computers in our shared labs are 15-year-old mini-PCs
running Windows 10; these were fine for the task, since
PrairieLearn adheres to widely-adopted (and therefore
legacy-compatible) standards for both 
the browser GUI and, importantly, accessibility.
For new CBTFs, we
purchased mini-PCs online for under \$500 each, fully equipped,
with Ethernet ports---we know better than to
rely on wi-fi for something so mission-critical in a densely-occupied room.

The computers are centrally managed.
Our shared labs use a campus-level Active Directory setup that allows
configuring different group policies (profiles) that allow blocking
software installation and filesystem access, locking out most of the
Internet, and so on.
A profile was created for CBTF use that 
disables USB ports (to prevent students bringing USB-borne materials
to an exam) and has virtually no software installed other than a Web
browser and some audio files of white noise for students
who want to block out audio distractions.
We use the open source Clonezilla to uniformly image all machines, and
DeepFreeze software is used to restore the computer to a specific
state whenever an account is signed out. This prevents the saving of
any data (including cookies, files, etc.) on the computer's local filesystem.


\subsection{Network: Not Just Firewalls}
\label{sec:networking}

\lesson{Delivering assessments securely is difficult if the LMS itself isn't
  designed to do so.}

Each lab has its own subnet, so the campus-level firewall was
configured to lock out most of the
Internet when accessed from one of the labs, except for the servers
needed to access PrairieLearn exams. 
One
caveat regarding this configuration was that attempting to access
any blocked site resulted in the same error message one would see if
there were no Internet connection at all.
Some students would make a typo in accessing PrairieLearn or
PrairieTest, see this error, and report to the proctor that their
computer had no Internet.  (This despite the fact that upon signing in
to the CBTF profile, there was an immediate and obvious desktop
shortcut that students were instructed to open to go directly to PrairieTest.)
An alternative would be to use Windows-level firewall facilities
rather than relying on a network firewall.

However, firewalling isn't enough to deliver secure exams.  
Specifically, our requirements for a secure exam are:

\begin{enumerate}

\item The exam can only be accessed while inside one of the CBTF
  spaces.

\item An exam can only be accessed by a particular student during
    the specific time of that student's reservation for that exam.  (That is, a
    student cannot make a reservation for exam A and use the
    opportunity to peek at exam B.)

 \item If the platform serving the exam is also the platform where
    homework assignments and other course materials are located,
      those other materials must not be accessible to the student
      while in the CBTF.

\end{enumerate}


Our campus LMS is Canvas, but even though it supports (impoverished)
``quizzes,'' it is not designed to deliver them securely.
While it supports item 1 by specifying CIDR blocks (IP subnets) to
which exam access is limited, supporting item 2 requires manually
editing each student's allowed exam time window in Canvas.
Furthermore, Canvas provides no way to restrict access to other
materials in the course space from within a particular CIDR
block---the only solution is to completely unpublish all other course
materials temporarily during a multi-day exam window, thereby making
them unavailable to all students during that window.

Some colleagues have reported using lockdown browsers such as
Respondus, but this leads to (at least) two problems, in addition to
the fundamental problem that such a browser is essentially spyware.
First, a lockdown browser must be installed and configured to work
with a particular set of applications (that is, a particular LMS), so
there is an $M\times N$ problem of licensing and installing a browser
to work with each LMS that could be used to administer exams.  Second,
if the LMS itself is not designed for secure delivery, careful
attention must be paid to avoid restricting access to secondary sites
that are actually needed for the exam, such as those that serve images
or provide an embedded coding environment, while blocking sites that
are truly unauthorized.  We judged this
Whac-A-Mole\footnote{Whac-A-Mole is a registered trademark (1975) of
Bob's Space Racers, Inc. Really.} to be unproductive and risky.

The correct solution is therefore a combination of a properly configured IP
firewall and an application that is actually
designed to deliver secure assessments (as defined by the above
requirements).  PrairieLearn is correctly designed for this use case.
The IP firewalling may be either host-level or CIDR-level, but is
independent of the application, unlike lockdown browsers.
At our institution, we created 
a CBTF-specific login profile whose password is publicly known.  That
profile is only available inside CBTF rooms, is configured with  the
necessary firewall rules to block browser access to any web site other
than PrairieLearn and our
campus SSO, and is the only profile from which in-CBTF exams are
accessible at all.


\subsection{Authentication: Dry Run It First}
\lesson{Two-factor authentication (2FA) can present subtle pitfalls in
  a CBTF environment.}

Most software systems integrated at our institution, including PrairieLearn, support
single sign-on (SSO) login via our Shibboleth-based institutional authentication
system.
However, as a matter of campus policy, most
SSO-accessible sites can trigger a 2FA flow%
%\footnote{\url{https://calnet.berkeley.edu/calnet-2-step}}
\footnote{\url{URL blinded for review}}
to complete login.
The most common way to do so is to respond to a challenge on one's
cell phone via a particular app (Cisco Duo in our case), but there are
other ways as well.  The 2FA requirement led to
various failure modes:

\begin{itemize}

\item Wifi and/or cellular data coverage is weak in the basement where the labs are, so the phone
  cannot complete the 2FA flow.
\item The student recently obtained a new cell phone and has not yet set
   up the 2FA app and linked it to campus.
\item The student normally uses a different method to complete 2FA,
  such as a physical USB token, tablet, or touchID-capable laptop,
  none of which are permitted in the CBTF.

\end{itemize}

The correct solution for us in our full-time CBTF is to disable 2FA
for logins on the CBTF's CIDR block.  But this is possible only with
close cooperation with campus IT (happily, the team responsible for
2FA and the team responsible for maintaining the shared instructional
labs are in the same campus unit) and because the full-time CBTF's
computers have their own CIDR block.  In one of our still-shared CBTF
spaces where we cannot disable 2FA, we allow students to have their
devices for the first 10 minutes of each exam slot in case needed to
complete login; at the 10-minute mark, before the proctor ``opens''
the exam in PrairieTest, all devices must be placed at the front of
the room.
